Game Engine Analysis (Godot)

by Kevin Vo

2017-05-07

Game Engine Analysis

As a developer starting off in the game development industry, one of the biggest challenges anyone faces is creating a game. This is already a huge task in itself, which is why most game developers often turn towards a game engine to help speed up this process instead of constructing their own game engine. There are many existing game engines today that provide the underlying framework and tools to create games, which means game developers have less to worry about and can instead focus on what’s more important; the final end product, their completed game.

Some of the most popular game engines today are Unity and Unreal Engine, both of which were paid software but have since recently been free to use (up to a certain point) (1). With games like Street Fighter V, Killing Floor 2 and XCOM being released on Unreal and games like Hearthstone, Cities Skylines and Firewatch being released in Unity it goes to show that these engines are in fact capable of producing high quality games for studios large and small. Unity and Unreal are considered more ‘feature complete’ game engines, in that they bundle almost all the aspects and features you would need to create your own game in their own engine.

There are several smaller/more barebone game engines that exist as well, such as GameMaker (2). These smaller game engines usually serve a different purpose or are less restricted in other areas in comparison to more feature complete engines. Complete game engines for instance let you create any game you wish as they include all the tools for you to do so, smaller game engines on the other hand are usually tailored towards a specific game style or game type so that those wishing to create a game tailored to a specific genre have more specific tools and features to do so. They also generally differentiate in some aspects from complete game engines such as licensing, cost and 2D/3D which gives game developers more options in terms of what game engine they should use.

Godot

Godot is a game engine that has recently started to grow in popularity, reaching 3rd place in GitHub’s game engine showcase with 8,000 stars (3). Godot was previously an in house game engine used by companies in America (4) but has since been open sourced to the community in 2014. Now, it is being actively developed on by the community and two of the original Godot creators, Juan Linietsky and Ariel Manzur to create a free, cross platform engine.

Godot aims to be a cross-platform game engine that is completely free and is community driven/developed (5). It is targeted towards 2D and 3D games however as the 3D aspect of Godot is still being developed over time, the 2D aspect is generally used by most of the community. To interact with Godot itself, a custom language called GDScript is used which has a custom syntax very similar to that of Python. The Godot team decided to use GDScript as they had encountered a lot of problems over the years with other scripting languages (6). Lua integration for example was extremely complex and took a lot of code, and Python bindings were difficult to embed into their own solution. With GDScript on the other hand, they were able to create the language in such a way that their design implementation of ‘nodes’ (covered later on) works and so the language is more tied into the game engine itself, so that it is native to Godot.

Godot was designed to be easy to use for newcomers and also for those who have used game engines before. The user interface is similar to Unity’s UI and also supports many features such as drag-and-drop functionality for nodes in the scene and live scene editing. By using GDScript, there are some initial barriers of entry but as it is similar to Python and is a scripting language, it is easy to dive into once the basics have been learned. Finally, as the editor itself is cross-platform (Windows, MacOS and Linux support), developers can work in the same editor across all their development environments.

Godot itself uses a node-based system for everything in the game. Whether it be scenes, objects or tiles inside your game, everything is a node which can have multiple child nodes and can inherit other nodes. This lets you add what you want to objects easily and intuitively. For instance, if your sprite/image node needs collision in the world, you can make the parent node a PhysicsBody which would then add collision to your sprite. With that, you can also add other nodes you have previously defined to new nodes to create new levels. For example, you could have a few house nodes and add it to a town node, and then later on add the town node to a village. Godot in essence is a tree of nodes that will become the game you wish to design, and you can reuse/instantiate any amount of nodes and the nodes you need to create the scene(s) you wish.

Godot comes with a custom inbuilt physics system that is tuned to work well in pixel-perfect 2D games (with kinematic and rigid bodies). There is also a built-in animation editor in the engine which lets you animate everything from properties on nodes to functions inside scripts as well. One of the animation methods is a cut-out system, whereby different parts of a sprite can be ‘cut-out’ and joined together in the animation editor with a skeleton. This skeleton can be used to link up all the parts of a sprite which then allows the developer to animate each body part individually. The body parts connected will then move together just like a real skeleton would (e.g. moving arms would also move the connected hand), making animating that much simpler for artists.

The best aspect of Godot however, is that it is also released under the MIT license and is open-source, which makes it completely free to use and edit! Many developers have been actively contributing to Godot’s source code to try and improve the game engine itself because of this, and as a result a new release of Godot (Godot 3.0) is planned to be released sometime later this year (8). Embracing the open source community by releasing Godot on an MIT license means that if there are any bugs within the game engine itself, or if the game engine needs to be modified then it can as all the developers have access to the source code for the engine itself on Github and thus can make a pull request to improve Godot in the future.

As of today, Godot contains a lot of different nodes that can be used in both 2D and 3D games. However, due to the complexity of 3D techniques there is always improvements that can be made to 3D technology, and thus 3D in Godot is in a good state but nowhere near the 2D support that Godot provides. Ranging from the documentation on the website to the example projects (8 example 3D projects vs 28 example 2D projects) located on Github, Godot is primarily used by developers as a 2D game engine. The website contains a lot of extensive documentation on all the aspects of the Godot engine itself, and tutorials on how to create a simple game in 2D which will aid developers wanting to get started in Godot and developers wanting to create a 2D game.

Finally, working as a team on a Godot project is actually quite simple as a result of how Godot was designed. As a Godot game/scene consists of a tree of nodes within nodes, different developers can work on their own nodes in the game which can then be instantiated later on into other nodes to populate a scene. Thus, while one developer is working on the Player node, another can work on the Inventory node and another on the actual Map node itself where the player will be fighting. This separation allows nodes to be worked on independently and re-used later on when needed. Different types of developers can also work in the project too, level designers can utilize the drag-and-drop functionality to create their own level designs and artists can utilize the animation editor to animate their sprites/models. And, when finally syncing files into a version control system like Git, everything just works as there are no binary files, everything in Godot is stored in a text file format which is friendly to VCS systems, meaning teams of any size can work together and create their dream game.

Godot is a new engine that is improving by the day. Being easy to get into with its high-level scripting language GDScript, learning how to create a simple game can be an easy exercise with the extensive documentation on the website. With the extensive features Godot offers alongside its ‘node’ system, Godot can be a good option for those looking for a good 2D engine to work in and maybe even complete a full game in.
